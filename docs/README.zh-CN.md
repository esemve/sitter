## 它是什么？

Sitter 是一个轻量级的 AI 原生工作流管理器，帮助你和你的 AI 编程代理进行结构化的开发周期。它将工作拆分为愿景、计划、任务和审查，所有这些都通过仓库中的 markdown 文件进行跟踪。

## 它解决了什么问题？

在使用大型代码库进行代理式开发时，我们面临以下几个问题：

- 需求偏离
- 反馈循环发生得太晚
- 实现不正确
- 无法预先创建足够详细的规格说明，而持续细化又极其耗时
- 生成大量代码，难以审查
- 审查困难，难以理解 LLM 决策背后的推理和动机

**Sitter 旨在帮助解决这些问题。**

使用 Sitter，你可以采用极早期反馈循环的开发方式。Sitter 将工作拆分为小任务，在代码中添加技术注释以说明实现背后的推理和决策，并在每个任务完成后要求进行审查。

在审查过程中，无论你是亲自修改代码还是让 AI 来修改，Sitter 都会重新规划剩余的任务并相应地更新总体计划。这防止了僵化地遵循前期计划，而是允许它根据审查过程中澄清的内容持续调整。

在实现过程中，AI 会在每处代码变更前添加注释，使 LLM 的推理过程透明化，简化审查过程，并允许尽早发现和纠正偏离。

## 安装

```bash
npm install -g @agentstuff/sitter
```

这将在全局安装 `sitter` CLI 命令。
```bash
sitter install
```

这将在你的 AI 代理中安装 Sitter 技能。

## 如何使用？

```bash
sitter init
```

这将在你的项目中初始化 `sitter/` 目录结构。
然后按顺序运行 Sitter 技能来管理你的项目：

1. `/sitter-vision` — 创建项目愿景。
2. `/sitter-brainstorm` — 通过提问来澄清需求。
3. `/sitter-plan` — 生成详细的实现计划和任务列表。
4. `/sitter-implement` — 逐个执行任务，每个任务完成后进行审查。
5. `/sitter-apply` — 应用审查后的变更并继续。
6. `/sitter-done` — 将已完成的项目存档。

| `/sitter-vision` | 创建项目愿景并初始化项目文件夹。生成 `vision.md` 文件。之后你可以在该文件中详细描述你的愿景——确切说明任务是什么以及你最终想达到的目标。 |
| `/sitter-brainstorm` | 提出澄清性问题并细化愿景文档。此命令基于 vision.md 文件工作。如果文件为空，它会不断向你提问，直到完全理解你想要构建什么。它主要从产品角度而非技术角度评估计划，并尝试识别潜在的边缘情况。 |

| `/sitter-implement` | 实现下一个待处理任务，将其标记为完成，并等待审查。它会找到下一个未完成的任务并实现它。完成后，会向用户请求审查。如果 `sitter/settings.yaml` 中的 `ai_comments` 值设为 true，它会在每处代码变更中添加前缀为 `@@AI@@:` 的注释，详细解释为什么需要该变更以及它做了什么。在审查阶段，如果任何要求的修改影响了计划或后续任务（例如重命名库或重构代码结构），它会根据新要求自动更新它们。如果你亲自修改了代码——比如编辑或重命名变量——请务必告知 AI。它会审查变更，并在必要时相应地更新计划和剩余任务。在审查阶段，你需要通过移除所有 `@@AI@@` 注释来证明你已经正确审查了代码。在此步骤完成之前，无法请求新的实现。 |
| `/sitter-yolo` | 连续实现所有剩余任务，仅在最后进行审查。 |
| `/sitter-apply` | 验证没有 AI 注释残留，并将项目从 REVIEW 阶段转移到 IMPLEMENT。你可以使用 Apply 命令接受 AI 的变更。重要的是，只有当代码库中 **没有任何** `@@AI@@` 注释时，才能请求 Apply。你需要通过从代码中移除所有此类注释来证明审查已完成。完成后，Apply 会重新启用 AI 继续实现，这意味着 `/sitter-implement` 再次可用，AI 可以开始下一批任务。 |
| `/sitter-done` | 将项目归档到存档文件夹并清理。 |

## @@AI@@ 注释

如果 `sitter/settings.yaml` 中的 `review/ai_comments` 设置为 `true`，那么在实现过程中，AI 必须在每处代码变更前添加 @@AI@@ 注释，详细解释为什么需要该变更、背后的技术推理以及它具体做了什么。目的是通过确保审查者——你——能够清楚理解 AI 在每个修改背后的决策过程，使审查阶段更加轻松。

在审查阶段，你需要通过移除所有 @@AI@@ 注释来证明你已经实际审查了代码。如果代码中任何地方仍然存在 @@AI@@ 注释，Apply 步骤将不被接受，因为这表明审查尚未正确完成。

如果你不喜欢这个功能，可以在 `settings.yaml` 中将其值设置为 `false`。

---

## TASK.md

`sitter` 目录中有一个 `TASK.md` 文件。在开始每个任务之前，AI 会读取此文件，并在实现过程中也考虑其中找到的指令。

它默认为空，但可用于为代理提供额外的实现特定指令。
