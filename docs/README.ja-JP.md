## Sitter とは？

Sitter は、AI コーディングエージェントと共に構造化された開発サイクルを進めるための、軽量な AI ネイティブワークフローマネージャーです。作業をビジョン、プラン、タスク、レビューに分割し、すべてリポジトリ内の Markdown ファイルで追跡します。

## どのような課題を解決するのか？

大規模なコードベースでエージェント型開発を行う際、以下のような課題に直面します：

- ドリフト（目的からの逸脱）
- 遅すぎるフィードバックループ
- 不正確な実装
- 事前に十分な詳細仕様を作成することは不可能であり、継続的に仕様を洗練させるのは非常に時間がかかる
- 生成されるコードが過剰で、レビューが困難
- LLM の判断の背後にある理由や動機を理解するのが難しい、困難なレビュー

**Sitter はこれらの問題の解決を目指します。**

Sitter を使うことで、非常に早い段階でのフィードバックサイクルを持つ開発ループを活用できます。Sitter は作業を小さなタスクに分解し、実装の背後にある判断理由を説明する技術コメントをコードに残し、各タスクの後にレビューを要求します。

レビュープロセス中に、コードを自分で修正する場合でも AI に修正を依頼する場合でも、Sitter は残りのタスクを再計画し、全体のプランをそれに応じて更新します。これにより、事前に立てた計画に固執することなく、レビュープロセスで明らかになったことに基づいて継続的に適応することが可能になります。

実装中、AI はすべてのコード変更の前にコメントを追加するため、LLM の判断理由が透明になり、レビューが容易になり、ドリフトを早期に検出して修正できます。

## インストール

```bash
npm install -g @agentstuff/sitter
```

これにより `sitter` CLI コマンドがグローバルにインストールされます。

```bash
sitter install
```

これにより Sitter スキルが AI エージェントにインストールされます。

## 使い方

```bash
sitter init
```

これにより、プロジェクト内に `sitter/` ディレクトリ構造が初期化されます。
その後、以下の Sitter スキルを順に実行してプロジェクトを管理します：

1. `/sitter-vision` — 新しいプロジェクトビジョンを作成します。
2. `/sitter-brainstorm` — 質問を通じて要件を明確化します。
3. `/sitter-plan` — 詳細な実装計画とタスクリストを生成します。
4. `/sitter-implement` — タスクを1つずつ実行し、毎回レビューを行います。
5. `/sitter-apply` — レビュー済みの変更を適用して続行します。
6. `/sitter-done` — 完了したプロジェクトをアーカイブします。

## スキル

| スキル | 説明                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
|-------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `/sitter-vision` | 新しいプロジェクトビジョンを作成し、プロジェクトフォルダを初期化します。`vision.md` を作成します。その後、このファイルにビジョンの詳細を記述できます — タスクの内容と、最終的に達成したいことを正確に記載します。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| `/sitter-brainstorm` | 明確化のための質問を行い、ビジョン文書を洗練させます。このコマンドは vision.md ファイルに基づいて動作します。ファイルが空の場合、構築したいものを完全に理解するまで質問を続けます。主に技術的観点ではなく製品の観点から計画を評価し、潜在的なエッジケースの特定を試みます。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| `/sitter-plan` | 詳細な `plan.md` を構築し、作業を `tasks.md` に分割します。vision.md に基づいて、技術仕様と一連の小さな論理的に連続したタスクを生成します。必要に応じてこれらを上書きできます。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| `/sitter-implement` | 次の未完了タスクを実装し、完了としてマークし、レビューを待ちます。次の未完了タスクを見つけて実装します。完了すると、ユーザーにレビューを要求します。`sitter/settings.yaml` の `ai_comments` の値が true に設定されている場合、すべてのコード変更に `@@AI@@:` というプレフィックス付きのコメントを追加し、その特定の変更がなぜ必要だったのか、何を行うのかを詳細に説明します。レビューフェーズ中に、要求された修正が計画や今後のタスクに影響を与える場合（例：ライブラリの名前変更やコード構造のリファクタリング）、新しい要件に従って自動的にそれらを更新します。自分でコードを修正する場合（例：編集や変数名の変更）、必ず AI にそのことを知らせてください。AI は変更をレビューし、必要に応じて計画と残りのタスクを更新します。レビューフェーズでは、すべての `@@AI@@` コメントをコード全体から削除することで、コードを適切にレビューしたことを証明します。このステップが完了するまで、新しい実装を要求することはできません。 |
| `/sitter-yolo` | 残りのすべてのタスクを連続して実装し、最後にのみレビューを行います。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| `/sitter-apply` | AI コメントが残っていないことを検証し、プロジェクトを IMPLEMENT に戻します。Apply コマンドで AI の変更を受け入れることができます。重要なのは、Apply はコードベースのどこにも `@@AI@@` コメントが **一切存在しない** 場合にのみ要求できるということです。コード全体からこれらのコメントをすべて削除することで、レビューが完了したことを証明します。これが完了すると、Apply によって AI が実装を続行できるようになり、`/sitter-implement` が再び利用可能になって、AI が次のタスクバッチの作業を開始できるようになります。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| `/sitter-done` | プロジェクトをアーカイブフォルダにアーカイブし、クリーンアップします。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |

## @@AI@@ コメント

`sitter/settings.yaml` で `review/ai_comments` が `true` に設定されている場合、実装中に AI はすべてのコード変更の前に @@AI@@ コメントを付ける必要があります。このコメントでは、その変更が必要だった理由、技術的な判断根拠、そして変更が具体的に何を行うのかを詳細に説明します。その目的は、レビュアー（あなた）が各修正の背後にある AI の意思決定プロセスを明確に理解できるようにすることで、レビューフェーズを容易にすることです。

レビューフェーズ中、すべての @@AI@@ コメントを削除することで、実際にコードをレビューしたことを証明します。コード内のどこかに @@AI@@ コメントが残っている場合、レビューが適切に完了していないことを示すため、Apply ステップは受け入れられません。

この機能が不要な場合は、`settings.yaml` で値を `false` に設定するだけです。

## TASK.md

`sitter` ディレクトリ内には `TASK.md` ファイルがあります。各タスクを開始する前に、AI はこのファイルを読み取り、実装中にそこに記載されている指示も考慮します。

デフォルトでは空ですが、エージェントに追加の実装固有の指示を提供するために使用できます。
